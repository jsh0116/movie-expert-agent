# SOLID 원칙

SOLID는 유지보수성과 확장성이 뛰어난 소프트웨어를 만들기 위한 5가지 기본 객체지향 설계 원칙입니다.

1. **S (Single Responsibility Principle, 단일 책임 원칙)**
   - 클래스는 단 하나의 책임만 가져야 하며, 클래스를 변경하는 이유는 단 하나여야 합니다.
   - 응집도를 높이고 유지보수를 쉽게 만듭니다.

2. **O (Open/Closed Principle, 개방-폐쇄 원칙)**
   - 소프트웨어 요소(클래스, 모듈, 함수 등)는 확장에 대해서는 열려 있어야 하지만, 변경에 대해서는 닫혀 있어야 합니다.
   - 기존 코드를 수정하지 않고도 새로운 기능을 추가할 수 있도록 다형성을 활용합니다.

3. **L (Liskov Substitution Principle, 리스코프 치환 원칙)**
   - 프로그램의 정확성을 깨뜨리지 않으면서, 기반 타입(base type)의 포인터/레퍼런스를 파생 타입(derived type)의 구조로 바꿀 수 있어야 합니다.

4. **I (Interface Segregation Principle, 인터페이스 분리 원칙)**
   - 클라이언트는 자신이 사용하지 않는 메서드에 의존하도록 강제되어서는 안 됩니다.
   - 목적과 용도에 맞게 인터페이스를 작고 구체적으로 분리해야 합니다.

5. **D (Dependency Inversion Principle, 의존성 역전 원칙)**
   - 고수준 모듈은 저수준 모듈에 의존해서는 안 되며, 둘 다 추상화에 의존해야 합니다.
   - 추상화는 세부 사항에 의존해서는 안 되며, 세부 사항이 추상화에 의존해야 합니다.
