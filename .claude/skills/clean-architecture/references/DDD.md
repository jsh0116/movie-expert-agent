# Domain-Driven Design (DDD) 가이드

도메인 주도 설계(DDD)는 복잡한 비즈니스 로직을 다루는 소프트웨어 시스템을 개발할 때, 복잡성을 낮추고 도메인 전문가와 개발자 간의 간극을 줄이기 위해 도입된 소프트웨어 설계 원칙입니다. Clean Architecture와 결합되어 가장 안쪽(Domain) 계층을 설계하는 기준으로 자주 활용됩니다.

## 핵심 요소 (Building Blocks)

* **Entity (엔티티)**
  - 고유한 식별자(ID)를 갖는 도메인 객체입니다. 속성이 바뀌더라도 동일한 객체로 취급됩니다.
  - 가급적 순수해야 하며 프레임워크나 외부 인프라에 의존해서는 안 됩니다.
  
* **Value Object (값 객체)**
  - 식별자가 없으며, 가지고 있는 속성값 전체가 객체의 정체성을 정의합니다.
  - 생성 후 변경되지 않는 불변(Immutable) 객체로 설계하는 것이 원칙입니다. (예: `Email`, `Address`, `Money`)

* **Aggregate (애그리거트)와 Aggregate Root**
  - 관련된 엔티티와 값 객체들의 군집을 의미하며 단위의 묶음입니다.
  - 데이터 변경이나 트랜잭션의 한 단위가 되며, 외부에서는 무조건 **Aggregate Root**를 통해서만 내부 데이터에 접근해야 합니다.

* **Repository (저장소)**
  - Aggregate의 컬렉션처럼 동작하며 영속성(Persistence) 처리를 추상화한 인터페이스입니다.
  - 도메인 계층에는 인터페이스(`Port`)만 정의하고, 실제 DB 코드는 인프라스트럭처 계층(`Adapter`)에 둡니다.

* **Domain Service (도메인 서비스)**
  - 엔티티나 값 객체의 행위로 만들기 모호하거나 애매한 비즈니스 로직을 처리하는 상태를 갖지 않는(stateless) 서비스입니다.
